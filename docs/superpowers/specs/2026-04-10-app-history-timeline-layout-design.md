# App 历史页时间轴型重构设计

## 1. 背景

当前历史页虽然已经完成协议迁移，并将当天明细从折线改为状态时间带，但页面整体仍然不好看，主要问题不在单个颜色或单个组件，而在页面层级与内容重心：

1. 顶部统计卡过扁，数值不够突出，像说明条而不是核心数据。
2. `7天趋势` 占位较大，但有效信息密度不高，底部日期 chip 反而比图更抢眼。
3. `当天明细` 已经换成状态时间带，但当前呈现仍偏细、偏空，不足以承担主图角色。
4. 异常记录列表过长且过密，导致页面视觉重心一路下坠，阅读节奏拖沓。

用户已明确反馈当前版本“还是很丑”，因此本轮不再继续做局部修饰，而是按页面级结构重排。

## 2. 目标

1. 将历史页改造成更清晰的“时间轴型”结构。
2. 让用户进入页面后，首先看到“今天的状态变化”，而不是次要统计或装饰性元素。
3. 保留 `7天趋势 = 曲线` 与 `当天明细 = 状态时间带` 这两个已确认方向，不再回退。
4. 在不改动历史数据来源和评分逻辑的前提下，显著改善视觉层级、可读性与页面节奏。

## 3. 非目标

1. 不改 OneNET 查询逻辑。
2. 不改 `historyChart.ts` 中已经验证通过的数据归一化与评分规则，除非页面重构确实需要最小范围的显示层辅助数据。
3. 不引入新的图表库。
4. 不把当天明细重新改回折线或柱状图。

## 4. 设计原则

### 4.1 以当天时间轴为主

历史页最有价值的信息不是“过去 7 天平均看起来如何”，而是“今天具体在什么时间段出现了哪些状态变化”。

因此页面主视觉应从 `7天趋势` 切换为 `当天明细`。

### 4.2 减少平均发光，保留重点发光

当前页面多个区块都在发光，导致没有真正的重点。本轮仅保留少量重点发光效果：

1. 关键数值
2. 当前选中态
3. 状态时间带中的高优先级状态色

其他区块改为更克制的深色面板与弱边框。

### 4.3 让版式承担层级，而不是让颜色硬撑层级

优先通过：

1. 模块顺序
2. 卡片高度
3. 留白
4. 字号对比
5. 区块密度

建立主次关系，而不是继续堆叠高饱和色或阴影。

## 5. 页面结构调整

### 5.1 顶部摘要区

保留三张统计卡，但整体改成更像“数据摘要卡”的布局：

1. 卡片高度增加，避免继续呈现为横向细条。
2. 标题放在上方小字，数值放在下方大字，确保数值成为视觉重心。
3. 三张卡仍为一行三列，但间距、内边距和纵向层级需要更明确。

三张卡的视觉主次：

1. `健康评分`：主卡气质，最亮，最强调数值。
2. `良好时长`：次级强调，偏冷色。
3. `异常次数`：警示色，但不应抢过 `健康评分`。

### 5.2 当天明细提升为第一主图

将 `当天明细` 区块移动到 `7天趋势` 之前，成为历史页的第一图表模块。

该模块应具备以下特征：

1. 面板高度明显高于 `7天趋势`。
2. 面板标题区整合：
   - 标题
   - 简短说明
   - 粒度切换
3. 面板内部保留状态时间带，但增强“时间轴”感，而不是“进度条”感。
4. 时间轴标签与轨道应形成完整的“日内状态面板”。

### 5.3 7天趋势降为辅助图

`7天趋势` 仍保留曲线，但从主图降为辅助图：

1. 面板高度缩短。
2. 底部大块日期 chip 改成更轻的刻度式布局。
3. 分数信息保留，但不再用厚重卡块承载。
4. 图例与数据来源标记弱化，避免喧宾夺主。

### 5.4 异常记录区收敛

异常记录区调整目标不是“做得更炫”，而是“不要拖垮整页节奏”：

1. 默认只显示近期一部分记录。
2. 增加“查看全部 / 收起”交互。
3. 单条记录卡更紧凑。
4. 时间标签保持醒目，但描述文案压缩为更短的辅文。

## 6. 关键模块设计

### 6.1 摘要卡设计

摘要卡应从“细长面板”改为“短高卡片”：

1. 增加纵向空间，让标题、图标、数值形成上中下层级。
2. 弱化每张卡的全边框发光。
3. 强化数值字号、字重、对比度。
4. `健康评分` 可使用更明显的数值高亮，但仍保持与现有主题一致。

### 6.2 当天明细时间轴设计

当天明细不是展示离散点，而是展示时间段状态变化，因此其视觉表达应接近“状态轨道”：

1. 背景轨道保持低对比度。
2. 正常段使用稳定绿色。
3. 异常段使用更醒目的暖色。
4. 无人段保留断开，不计入评分。
5. 轨道厚度适当增加，让主图更有存在感，但不能变回大粗条。
6. 补充轻量网格或时段辅助线，增强时间感。

标题区可以直接补充一个当天摘要，如：

1. 今日活跃时长
2. 今日异常次数
3. 当前粒度说明

但这些信息应是辅助，不应再次堆成独立大卡。

### 6.3 7天趋势设计

7天趋势的作用是让用户快速判断整体走势，因此应强调“轻”和“快读”：

1. 曲线保留。
2. 节点保留，但缩小。
3. 底部使用更轻的日期刻度和小分数标注。
4. 面板整体高度小于当天明细。
5. 选中某天时仍支持切换下方当天明细内容。

### 6.4 异常记录设计

异常记录改为“补充证据区”，而不是主内容区：

1. 记录卡更薄、更紧凑。
2. 图标区缩小。
3. 时间标签保留胶囊样式，但整体更克制。
4. 首屏只给有限数量，避免页面无限下坠。

## 7. 交互设计

1. `7天趋势` 仍可点击不同日期，驱动当天明细与记录同步切换。
2. `当天明细` 的粒度切换仍保留 `5/10/15 分钟`。
3. 异常记录增加“展开/收起”切换。
4. 当无有效当天数据时，空状态文案保留，但空状态容器应更像信息面板而不是占位块。

## 8. 数据与实现边界

本轮优先改 `app/src/pages/history/index.vue` 的页面结构与样式；允许在以下范围内做最小逻辑补充：

1. 增加记录列表展开/收起所需的本地状态。
2. 增加当天摘要展示所需的派生字段。
3. 若 `7天趋势` 底部信息改造需要更轻量的显示数据结构，可在当前页面内做最小派生，不外溢到共享 store。

不做新的全局状态抽象。

## 9. 验证方式

### 9.1 视觉验证

1. 在 H5 下查看历史页桌面视口。
2. 在 H5 下查看历史页移动端视口。
3. 确认页面主次顺序变为：
   - 摘要卡
   - 当天明细
   - 7天趋势
   - 异常记录

### 9.2 功能验证

1. 选中不同日期时，当天明细和异常记录仍能同步切换。
2. 粒度切换仍正常工作。
3. 无人数据仍保持断开，不计入评分。
4. 异常记录展开/收起行为正常。

### 9.3 工程验证

1. 运行 `npm run type-check`
2. 运行 `npm test`

## 10. 影响范围

主要文件：

1. `app/src/pages/history/index.vue`
2. `app/src/components/ui/StatCard.vue`（若需要增强摘要卡的表现力）

不计划修改协议层与历史查询接口。
