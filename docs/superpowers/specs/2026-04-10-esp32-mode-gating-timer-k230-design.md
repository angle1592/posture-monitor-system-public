# ESP32 模式独占、K230 停检与定时器提醒设计

## 1. 背景

在上一轮 `ESP32 融合姿态、OLED 与补光` 设计落地后，系统已经具备：

- 基于 `PIR + K230 posture_type` 的融合姿态结果
- OLED 三模式页面
- 基于环境光的补光控制
- 本地定时器功能

但实机联调暴露出新的交互与状态管理问题：

1. OLED 顶部 `WiFi/MQTT` 状态栏占用了大字页面的垂直空间，导致时钟页和定时器页与横线重叠。
2. `时钟模式` 和 `定时器模式` 目前只是页面切换，K230 检测、补光和坐姿提醒仍在后台继续运行。
3. 定时器设置态与倒计时运行态耦合不清，导致“设置态启动后仍停留在设置态”和“暂停后无法再次设置时间”等问题。
4. 定时器结束后只有一次性提醒，未形成“直到用户确认才解除”的完整提醒链路。

本设计将当前系统改为“单模式独占运行”，并引入 `ESP32 -> K230` 的最小下行控制协议，确保模式语义、提醒语义和硬件行为一致。

## 2. 目标

本次设计的目标是：

1. 将系统运行时改为单模式独占：切到某个模式后，其它模式不在后台继续运行。
2. 为 `ESP32 -> K230` 新增最小 UART 控制命令，让 `时钟/定时器/检测关闭` 场景下 K230 真正停检。
3. 重做 OLED 顶部布局，去掉 `WiFi/MQTT` 状态栏，避免时钟页和定时器页大字与横线重叠。
4. 修正定时器的设置态/运行态/结束确认态交互，使 EC11 行为符合实际使用预期。
5. 将“定时器结束提醒”纳入与坐姿提醒相同的提醒模式掩码体系，并建立明确的提醒优先级。

## 3. 非目标

以下内容不在本次范围内：

1. 不修改 K230 坐姿识别算法和阈值本体。
2. 不在本次引入复杂的 K230 资源释放/重建流程；K230 仅实现“软停检”。
3. 不新增独立的云端事件模型，继续通过现有属性模型表达运行状态。
4. 不重构现有 `mode_manager`/`timer_fsm` 的整体架构，只做完成目标所需的最小扩展。

## 4. 单模式独占运行

### 4.1 设计原则

系统任一时刻只允许一个主模式处于活动态：

- `MODE_POSTURE`
- `MODE_CLOCK`
- `MODE_TIMER`

切换到某一模式时，其他模式的业务逻辑必须停止，而不是仅仅停止显示。

### 4.2 各模式的运行语义

#### 坐姿检测模式

当 `currentMode == MODE_POSTURE` 且 `monitoringEnabled == true` 时：

- K230 处于检测开启状态
- ESP32 允许读取并消费 K230 姿态结果
- 姿态融合、补光控制、坐姿提醒正常运行
- 定时器不在后台运行

#### 时钟模式

当 `currentMode == MODE_CLOCK` 时：

- K230 停止检测
- ESP32 停止坐姿融合、补光和坐姿提醒
- 定时器不运行
- OLED 只显示时钟页面

#### 定时器模式

当 `currentMode == MODE_TIMER` 时：

- K230 停止检测
- ESP32 停止坐姿融合、补光和坐姿提醒
- 定时器是唯一活动业务
- OLED 只显示定时器页面

### 4.3 与 monitoringEnabled 的关系

`monitoringEnabled` 仍保留为云端/本地的“坐姿检测开关”，但其生效条件为：

```text
posture_pipeline_active = (currentMode == MODE_POSTURE) && monitoringEnabled
```

因此：

1. 切到 `时钟/定时器` 模式时，即使 `monitoringEnabled=true`，坐姿链路仍然停用。
2. 位于 `坐姿检测模式` 但用户或云端关闭 `monitoringEnabled` 时，也应向 K230 下发停检命令。

## 5. ESP32 -> K230 下行控制协议

### 5.1 设计目标

K230 不应仅因 ESP32 本地忽略结果而继续推理；切到非坐姿模式或关闭检测时，应由 ESP32 显式通知 K230 停止检测。

### 5.2 最小命令格式

ESP32 向 K230 发送逐行 JSON 命令：

```json
{"cmd":"set_detection","enabled":false}
```

字段定义：

- `cmd`
  - 固定值：`set_detection`
- `enabled`
  - `true` 表示恢复检测
  - `false` 表示停检

### 5.3 ESP32 行为

ESP32 应维护一个“期望的 K230 检测开关状态”，仅在状态变化时发送命令，避免无意义重复写串口。

状态映射如下：

- `posture_pipeline_active = true` -> 发送 `enabled=true`
- `posture_pipeline_active = false` -> 发送 `enabled=false`

### 5.4 K230 行为

K230 增加一个运行态标志：

- `detection_enabled = True/False`

当 `detection_enabled == False` 时：

1. 不执行 `capture -> process_frame -> posture output` 主路径。
2. 不向 ESP32 发送姿态 JSON。
3. 仍保留主循环和 UART 接收能力，用于等待后续恢复命令。

该设计选择“软停检”而非释放摄像头/NPU 资源，是为了降低恢复路径风险，保证切回坐姿模式时可快速恢复。

## 6. OLED 版式调整

### 6.1 设计目标

OLED 页面不再把 `WiFi/MQTT` 视为最重要的信息，而应优先保证：

1. 时钟页和定时器页的大字区域完整可见。
2. 页面主题明显，视觉层级清楚。
3. 不同模式之间的布局具有一致的头部语言。

### 6.2 通用布局原则

所有页面统一采用：

1. 顶部标题
2. 装饰性横线
3. 主内容区
4. 底部辅助信息区

禁止再绘制 `WiFi/MQTT` 状态栏，以避免侵占大字页面空间。

### 6.3 各页面布局

#### 坐姿页

- 标题：`坐姿监测`
- 主内容：`当前姿态: 正常/低头/驼背/无人`
- 辅助信息：`人体`、`补光`、`光照`

#### 时钟页

- 标题：`时钟`
- 主内容：大号 `HH:MM:SS`
- 辅助信息：日期

要求大号时间整体下移，确保与顶部横线不重叠。

#### 定时器页

- 标题：`定时器`
- 主内容：大号 `MM:SS`
- 辅助信息：当前状态文本

状态文本包括：

- `准备开始`
- `倒计时中`
- `已暂停`
- `时间到了`
- `调时中`

## 7. 定时器状态与交互设计

### 7.1 状态划分

定时器扩展为以下状态：

- `IDLE`
- `RUNNING`
- `PAUSED`
- `DONE_PENDING`

其中 `DONE_PENDING` 表示“倒计时结束，但用户尚未确认提醒”。

### 7.2 EC11 交互规则

#### 长按

在 `定时器模式` 中：

1. 长按一次：进入设置态
2. 再长按一次：退出设置态

但在 `DONE_PENDING` 期间，长按无效。

#### 旋转

仅在“设置态”下用于调节时间。

#### 短按

在 `定时器模式` 中：

1. `IDLE`
   - 短按开始倒计时
   - 若此时处于设置态，则开始后自动退出设置态

2. `RUNNING`
   - 短按暂停

3. `PAUSED`
   - 短按恢复

4. `DONE_PENDING`
   - 短按表示“确认时间到了”
   - 确认后清除结束提醒，恢复为 `IDLE`

### 7.3 设置态时间调整语义

#### IDLE / DONE_PENDING 之后进入设置态

调整的是“默认倒计时时长”。

#### PAUSED 进入设置态

调整的是“当前暂停会话的剩余倒计时时间”，并立即反映到页面显示。

这样可修复“暂停后长按进入设置态但无法改时间”的问题。

### 7.4 模式切换约束

1. 只要定时器处于 `DONE_PENDING`，就不允许切换模式。
2. 用户必须先短按确认“时间到了”，模式切换才重新开放。
3. 一旦离开 `定时器模式`，定时器相关运行状态应整体停止并回到空闲态，不允许在后台继续运行。

## 8. 定时器结束提醒

### 8.1 设计目标

定时器结束后应像闹钟一样进入“间歇提醒，直到确认”的状态，而不是只提醒一次。

### 8.2 提醒触发规则

当倒计时从 `RUNNING` 到达结束点时：

1. 进入 `DONE_PENDING`
2. 立即触发一轮“时间到了”提醒
3. 之后每隔固定若干秒重复提醒一次
4. 直到用户短按确认才停止

### 8.3 提醒输出通道

定时器结束提醒复用现有 `alertModeMask`：

- `LED`
- `BUZZER`
- `VOICE`

这意味着用户在云端对提醒模式的配置同时影响坐姿提醒和定时器结束提醒。

### 8.4 语音文案

定时器结束提醒语音采用更明确的时间提醒文案，例如：

- `时间到了，时间到了`

### 8.5 与坐姿提醒的优先级

提醒优先级固定为：

```text
定时器结束未确认 > 坐姿提醒
```

因此：

1. 只要处于 `DONE_PENDING`，坐姿提醒必须被完全压制。
2. 用户确认定时器结束提醒后，坐姿提醒能力才恢复。

在本设计的“单模式独占”前提下，`DONE_PENDING` 期间本就位于 `定时器模式`，因此不会与实时坐姿提醒同时触发；但该优先级仍需在提醒执行层明确保留，防止未来状态扩展时出现竞态。

## 9. 实现边界

本次实现涉及以下活跃子树：

- `posture_monitor`
  - 模式管理、OLED、定时器、提醒、K230 串口发送控制
- `k230/src`
  - UART 下行命令接收与 `detection_enabled` 软停检控制

不需要修改共享协议常量文件，因为本次新增的是设备内部控制命令，不进入 OneNET 物模型。
