# App 开发态 Mock 历史与实时数据设计

## 1. 背景

历史页已经切换为：

- `7 天趋势贝塞尔曲线`
- `当天明细贝塞尔曲线`
- `no_person` 断线
- `5/10/15 分钟` 聚合切换

但当前真实设备历史数据不足，无法稳定验证：

1. 周趋势高低分变化
2. 当天明细多段曲线
3. `no_person` 断线效果
4. 实时页多姿态状态切换

因此需要一个**仅用于开发预览**的统一 Mock 数据源，用于快速喂满首页、实时页和历史页。

## 2. 目标

本次设计目标：

1. 通过环境变量一键切换 `真实 OneNET` / `Mock 数据源`
2. 让首页、实时页、历史页都能从同一套 mock 数据获取内容
3. 保证 mock 数据覆盖：
   - `normal`
   - `head_down`
   - `hunchback`
   - `no_person`
4. 让历史页在 mock 模式下能稳定展示：
   - 7 天曲线
   - 当天 5/10/15 分钟聚合曲线
   - 断线段
   - 异常记录

## 3. 非目标

1. 不向 OneNET 回写 mock 数据
2. 不伪造真实云端历史时间戳
3. 不在正式 UI 增加“启用测试数据”入口
4. 不影响控制页真实下发逻辑的最终实现方向

## 4. 开关方式

使用环境变量控制：

```env
VITE_USE_MOCK_HISTORY=true
```

语义：

- `true`：App 读 mock 实时/历史数据
- `false`：App 读真实 OneNET 数据

该开关仅服务开发态，不应进入生产环境配置。

## 5. 模块边界

### 5.1 新增模块

建议新增：

- `app/src/utils/mockData.ts`

职责：

1. 生成 mock 实时属性流
2. 生成 mock 历史属性流
3. 保证三页共享同一姿态语义

### 5.2 接入位置

建议只在 `oneNetApi.ts` 这一层接入 mock 分流：

1. `queryDeviceProperty()`
2. `queryPropertyHistory()`
3. `queryDeviceStatus()`

当 `VITE_USE_MOCK_HISTORY=true` 时，这三个入口直接返回 mock 数据。

这样：

- `store.ts` 不需要改调用方式
- 页面层不需要知道当前是真数据还是 mock 数据
- 可以最大限度复用现有数据流和页面逻辑

## 6. Mock 数据内容

### 6.1 实时数据

mock 实时数据需要包含：

- `postureType`
- `personPresent`
- `fillLightOn`
- `monitoringEnabled`
- `currentMode`

实时姿态建议按轮询时序循环切换，例如：

1. `normal`
2. `head_down`
3. `hunchback`
4. `no_person`

这样刷新实时页时能直接观察到四态变化，而无需反复改代码。

### 6.2 历史数据

mock 历史数据建议覆盖最近 7 天，并满足：

1. 每天都有不同健康分布
2. 至少部分时间段为 `no_person`
3. 同一天内既有正常区段，也有低头/驼背区段

### 6.3 当天数据密度

当天 mock 历史应按足够高的密度生成原始点，以支持：

- `5 分钟` 聚合
- `10 分钟` 聚合
- `15 分钟` 聚合

建议直接按 `5 分钟` 间隔生成底层历史点，再由现有聚合函数折算更粗粒度。

## 7. 页面效果预期

启用 mock 后：

### 首页 / 实时页

- 可以轮流看到：
  - 正常
  - 低头
  - 驼背
  - 无人

### 历史页

- 7 天曲线不再是单一平线
- 当天明细曲线能够看到明显波动
- `no_person` 时间段显示为断线
- 异常记录列表有稳定内容可供预览

## 8. 风险与约束

1. mock 数据必须遵守当前活跃协议语义，不能再回退为 `isPosture` 二值模型。
2. mock 数据应稳定可复现，避免每次刷新都完全随机导致 UI 调试困难。
3. mock 模式不能污染真实请求逻辑；开关关闭后应完全走回 OneNET。
